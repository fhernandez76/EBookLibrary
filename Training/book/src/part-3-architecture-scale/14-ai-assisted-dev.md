## 14.1 An honest accounting

This book — its prose and its code — was built with substantial AI
assistance. The Domain entities of Chapter 4 were drafted by Claude
and revised by hand. The MediatR pipeline of Chapter 5 was first
generated by Copilot and then audited line by line for the
single-responsibility violation any LLM is prone to. The React
frontend of Chapter 10 was about 60% AI-drafted, 40% hand-written, by
volume — and the *important* 5% (auth state, error handling, optimistic
update rollback) was almost entirely hand-written, because that is
where AI's defaults silently miss.

The chapter is about what those numbers mean. There is a productive
collaboration to be had with modern coding assistants. There is also
a sloppy one. The difference is mostly habits.

> **Architect's Note:** The headline-grabbing "AI writes 80% of the
> code" stat from various surveys is an arithmetic average that hides
> a bimodal distribution. The first mode is "boilerplate" — DTO
> mappers, repetitive controller methods, test scaffolds — where AI is
> very nearly 100% productive. The second mode is "load-bearing logic"
> — security, concurrency, schema, public API surface — where AI is
> better thought of as a reviewer than an author. Treat them
> differently.

## 14.2 Context curation, the only thing that really matters

Across every assistant, every model, every workflow, the *single*
pattern that distinguishes good output from bad is **what context
the model had**. The same model, given the project's `IUnitOfWork`
interface, the relevant entity, and a clear prompt, produces a
correct handler. The same model, given just the prompt, produces a
plausible handler that violates conventions established three folders
away.

The implication: most of your job in an AI-assisted workflow is *not*
"write a clever prompt". It is "make sure the model can see the right
files".

Three concrete tactics that pay back the time:

1. **Open the files that demonstrate the conventions** before
   prompting. In Cursor and Copilot Chat, open files become context.
   In Claude or ChatGPT, paste the relevant snippets explicitly.
2. **Share the *interface*, not the implementation.** When asking
   for a new handler, share `IBookRepository`, `IUnitOfWork`,
   `Result<T>` — not the existing handler implementations. The model
   should write the implementation; the interfaces tell it what is
   available.
3. **Share one *example* of the pattern.** A single existing handler
   that establishes "this is what handlers look like in this project"
   is worth more than a paragraph of prose describing the same.

## 14.3 The four prompt patterns

Across hundreds of interactions, four patterns kept producing better
code than ad hoc prompting.

### Pattern 1 — Role + Context + Constraints + Example

```text
You are a senior .NET 10 backend engineer working in a Clean
Architecture solution. (role)

The project uses MediatR for CQRS, FluentValidation for validation,
and a Result<T> envelope returned from every handler.

Existing interfaces (paste IUnitOfWork, IBookRepository, Result<T>).
Existing handler example: CreateBookCommandHandler (paste).

(context)

Constraints:
- No exceptions for business validation; use Result<T>.Failure.
- Use the soft-delete-aware repository methods (no .IgnoreQueryFilters).
- The handler must be sealed.

(constraints)

Task: write UpdateBookCommandHandler that updates an existing book's
title, ISBN, and description. Keep the same shape as
CreateBookCommandHandler.

(example-anchored task)
```

This pattern works because it removes ambiguity. The model is not
guessing what the project's idioms are; it is matching them.

### Pattern 2 — Plan first

For any change touching more than one file, ask the model for a plan
*before* it writes code:

```text
Before you write any code, give me a numbered plan: which files you
will create, which you will modify, and the rough shape of each
change. Wait for my approval before writing the code.
```

Two-thirds of the time the plan exposes a misunderstanding that would
have wasted ten minutes of generated code. The other third you say
"yes" and the work proceeds.

### Pattern 3 — Explain back to me

After a complex piece of logic — a query, a security check, a state
machine — ask:

```text
Walk me through what this code does, line by line, and identify any
edge cases that would break it.
```

If the explanation reveals the model "thinks" the code does
something the code does not, that is your bug detector firing.
Sometimes the code is right and the explanation is wrong (rare);
sometimes the explanation is right and the code is wrong; usually
both are wrong in the same direction.

### Pattern 4 — Constrain the diff

For edits to existing files, constrain *which* parts can change:

```text
Modify only the SearchAsync method. Do not change any other method
signature. Preserve all using-statements. Show the diff, not the full
file.
```

Without this constraint, models cheerfully "improve" unrelated code
and quietly delete edge cases your tests covered. The diff form
makes drift visible.

## 14.4 The bugs to watch for

AI assistants introduce a recognizable family of bugs. The unit tests
of Chapter 12 are specifically designed to catch them; if you know
what to look for, you can short-circuit the cycle.

**Table 14.1 — AI-introduced bug families.**

| Family                          | Example                                                               | Who catches it                                |
|---------------------------------|-----------------------------------------------------------------------|-----------------------------------------------|
| Plausible-but-wrong API names   | `BCrypt.HashPassword(p, 12)` vs. `BCrypt.Net.BCrypt.HashPassword`     | Compiler                                      |
| Outdated framework idioms       | `services.AddSwaggerGen()` in a project using Scalar (§ 7.8)          | Code review                                   |
| Defensive-programming bloat     | `if (book == null) throw…` for a value the type system already proves | Code review                                   |
| Silent SR violation             | A handler that also does file I/O (§ 5.7)                             | Architect's eye                               |
| Off-by-one / sign errors        | `Skip((page) * size)` instead of `Skip((page - 1) * size)`            | Unit test for paging                          |
| Forgotten `Times.Never`         | Negative test asserts the error but does not assert the *non-save*    | Mutation test                                 |
| Wrong claim type (§ 8.4)        | `new Claim("role", ...)` instead of `ClaimTypes.Role`                 | The integration test in Listing 12.3          |
| Missing CORS preflight handling | Auth middleware before CORS (§ 7.5)                                   | The first browser-driven E2E (Chapter 13)     |
| Hallucinated NuGet packages     | `Microsoft.AspNetCore.SomethingPlausible`                             | NuGet restore                                 |

The remedy in every row is the same: a test, a compiler, or a human
review. The lesson is that an AI-assisted workflow is not "AI writes
the code" — it is "AI writes the code, the *system* catches the
mistakes".

> **Pitfall:** "I'll skip the tests because Copilot is so reliable
> these days" is the single most expensive false economy in modern
> software. Copilot's reliability *increases* the value of your
> tests, because the tests are the only thing standing between you
> and the bug class above. Write the tests first; let the AI write
> the implementation; let the tests catch what the AI missed.

## 14.5 Three real workflows

The three workflows below are the ones used in this project. Each
has a sweet spot.

### Copilot inline (autocomplete)

Best for: the last 30% of a method you have already started writing.
"Type the method signature and let it finish" is where Copilot inline
shines.

Worst for: load-bearing logic. The autocomplete suggestion is
optimized for plausibility, not correctness. Read every accepted
completion before tabbing.

### Copilot Chat / Agent Mode

Best for: scaffolding new files in a project with established
conventions. "Create a new MediatR handler for X" with the existing
handler open as context produces near-perfect output.

Worst for: cross-cutting changes. Agent mode will happily edit
fifteen files; auditing fifteen files of AI-generated change is *not*
faster than writing the changes yourself.

### Claude (or any LLM) in a separate window

Best for: design conversations. "Here is my proposed schema; what
will this be hard to do in a year?" is the kind of question a chat
window with a powerful model answers very well.

Worst for: tight code-iteration loops. The copy-paste tax kills
productivity for "make this small change to that file"; let Copilot
do those.

> **In Practice:** The workflow this book was written under was a
> *combination*: Claude in a chat window for chapter outlines and
> design discussion, Copilot inline for code completion within a
> chapter's listings, Copilot Chat / Agent Mode for "scaffold the
> next chapter file using the established structure". No one tool
> would have produced the result alone.

## 14.6 The economics

A practical way to think about AI's impact on a project of this
scale.

- **Boilerplate** (DTO mappers, repetitive validators, test
  scaffolds, Razor markup): the LLM is 5–10× faster than typing.
- **Pattern-matching to existing conventions** (a new handler in the
  shape of an existing one, a new admin page in the shape of an
  existing one): 3–5× faster.
- **Load-bearing original logic** (auth flow, schema design,
  concurrency, public API contract): roughly the same speed as
  hand-written, with a *higher* defect rate if accepted blindly.
- **Documentation** (this book): 2–3× faster, with a heavier review
  burden than code because prose has fewer automatic correctness
  checks.

The aggregate productivity win is real and substantial. The
distribution is not uniform. Plan accordingly.

## 14.7 A short ethical postscript

LLMs are trained on code that includes vulnerabilities, dead idioms,
and license-restricted snippets. The two practices that cover most
of the risk:

1. **Run a license scanner** (e.g., `licensee`, the GitHub-built one)
   on AI-generated code that you commit. The risk of an MIT-licensed
   project ingesting a GPL-licensed snippet by accident is real and
   small; the scanner catches the obvious cases.
2. **Re-read security-relevant code by hand.** Auth, crypto, file
   handling, deserialization — never accept these without
   line-by-line review.

> **Architect's Note:** "The model produced the bug" is not a defense
> in a security incident. The accountability stays with the human who
> committed the code. Adopt AI tools with that frame and the
> incentives stay aligned.

## Key takeaways

- Context curation is the central skill. The model can only do well
  what it can see clearly.
- Four prompt patterns repeatedly produce better code: *role +
  context + constraints + example*, *plan first*, *explain back to
  me*, *constrain the diff*.
- AI introduces a recognizable family of bugs. The unit, integration,
  and E2E tests of Chapters 12 and 13 are specifically the line of
  defense.
- Three workflows — inline autocomplete, agent mode, separate-window
  chat — each have a sweet spot. Use them in combination.
- Accountability for AI-generated code rests with the human who
  commits it. Especially for security-relevant code, re-read by
  hand.

## Exercises

**Easy.** Pick one method you wrote by hand earlier in the book and
ask the AI to "explain back to you" what it does line by line. Note
any places the explanation is wrong.

**Medium.** Write a fresh MediatR handler from scratch using the
*Role + Context + Constraints + Example* pattern. Compare the output
to one written without the pattern. Score both for: convention
adherence, defect count after review, time-to-correct.

**Hard.** Build your team's *prompt library* — five reusable prompts
for the most common operations in your codebase (new handler, new
endpoint, new test, refactor for a specific pattern, code review).
Commit the library to the repo's `docs/prompts/` folder. Discuss
what changes when prompts become a versioned project artifact.
