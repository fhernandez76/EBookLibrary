## 26.1 The three packages

The i18next ecosystem is split across three packages.

| Package | Role |
|---|---|
| `i18next` | Core library. Manages resources, detects language, and runs the `t()` translation function. Framework-agnostic. |
| `react-i18next` | React integration. Provides the `useTranslation` hook, the `<Trans>` component, and connects i18next to React's re-render cycle. |
| `i18next-browser-languagedetector` | Automatically detects the user's language from the browser's `navigator.language`, a query string (`?lng=es`), a cookie, or `localStorage`. |

You installed all three in Chapter 17:

```powershell
npm install i18next react-i18next i18next-browser-languagedetector
```

## 26.2 The initialisation file

**File:** `src/i18n/index.ts`

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';

import en from './locales/en.json';
import es from './locales/es.json';

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: {
      en: { translation: en },
      es: { translation: es },
    },
    fallbackLng: 'en',
    interpolation: {
      escapeValue: false,   // React already escapes output
    },
  });

export default i18n;
```

The `.use()` calls register plugins with the i18next core. The order
matters:

1. `LanguageDetector` must come before `initReactI18next` so the
   language is resolved before React reads it.
2. `initReactI18next` wires the core library to React's context so
   that `useTranslation()` works inside any component.

`escapeValue: false` prevents i18next from double-escaping HTML
entities. React already escapes strings before rendering them,
so i18next's own escaping is redundant.

## 26.3 Import before the app renders

The `i18n/index.ts` file must be executed before any component
renders. In `src/main.tsx`, the import happens at the top of the
file before the `ReactDOM.createRoot()` call:

```tsx
import './i18n';          // ← must be first
import React from 'react';
import ReactDOM from 'react-dom/client';
import { BrowserRouter } from 'react-router-dom';
import { QueryClientProvider } from '@tanstack/react-query';
import App from './App';
import './index.css';
```

:::warning
**Order matters.** If `./i18n` is imported after any component
import, the translations will not be available the first time a
component renders and you will see the raw translation keys
(e.g., `nav.home`) instead of translated strings.
:::

## 26.6 Using translations in components

### The `useTranslation` hook

Every component that needs translated text calls `useTranslation()`.
It returns a `t` function and the `i18n` instance:

```tsx
import { useTranslation } from 'react-i18next';

function MyComponent() {
  const { t, i18n } = useTranslation();

  return <h1>{t('home.hero_title')}</h1>;
}
```

The `t()` function takes the dot-notation key from the JSON file and
returns the string for the current language.

### String interpolation

Use double curly braces `{{variable}}` in the JSON string, then
pass a values object as the second argument:

```tsx
// Translation file:
// "pages": "{{count}} pages"
// "page_of": "Page {{current}} of {{total}}"

t('book.pages', { count: 450 })     // → "450 pages"
t('common.page_of', { current: 2, total: 10 })   // → "Page 2 of 10"
```

## 26.7 Changing language programmatically

`i18n.changeLanguage(code)` changes the language at runtime. All
components that use `useTranslation()` re-render automatically.

The language is persisted to `localStorage` by the `LanguageDetector`
plugin. The next time the user loads the page, the detector reads it
from storage and restores their preference.

In `PublicLayout`, the toggle is already wired:

```tsx
const { i18n } = useTranslation();
const toggleLang = () => i18n.changeLanguage(i18n.language === 'en' ? 'es' : 'en');
```

## 26.8 LanguageSwitcher component

This is a standalone button that can be placed anywhere in the app.

**File:** `src/components/LanguageSwitcher.tsx`

```tsx
import { useTranslation } from 'react-i18next';
import { Globe } from 'lucide-react';

export default function LanguageSwitcher() {
  const { i18n } = useTranslation();
  const isEnglish = i18n.language === 'en';

  return (
    <button
      onClick={() => i18n.changeLanguage(isEnglish ? 'es' : 'en')}
      className="flex items-center gap-1.5 text-sm text-gray-600 hover:text-primary-500 transition-colors"
      title={isEnglish ? 'Cambiar a Español' : 'Switch to English'}
    >
      <Globe className="w-4 h-4" />
      <span className="font-medium">{isEnglish ? 'ES' : 'EN'}</span>
    </button>
  );
}
```

The button label shows the *target* language (where the user will go
when they click), not the current language. This is the standard UX
convention.

## 26.9 The Suspense pitfall

When i18next loads translations asynchronously (from a remote server
or a backend plugin), it is not ready on the first render. In this
project the translations are bundled inline (imported as JSON), so
they are synchronously available.

If you later switch to loading translations from an HTTP endpoint,
you must wrap your app in `<Suspense>`:

```tsx
import { Suspense } from 'react';
// ...
<Suspense fallback={<div>Loading...</div>}>
  <App />
</Suspense>
```

Without `Suspense`, the app will throw an unhandled error when
`useTranslation()` is called before the resources are loaded.

:::tip
For this project you do **not** need `Suspense` because translations
are bundled. Keep this note in mind if you add a backend translation
service later.
:::

## Key takeaways

- Import `./i18n` before any component import in `main.tsx`.
- `t('key', { variable: value })` handles string interpolation with
  double-brace syntax in the JSON files.
- `i18n.changeLanguage('es')` triggers an immediate re-render across
  every component that uses `useTranslation()`.
- `LanguageDetector` persists the user's choice to `localStorage`
  automatically.
- Set `escapeValue: false` — React already escapes output, so
  double-escaping creates garbled strings.
- `Suspense` is only needed when translations are loaded
  asynchronously (HTTP). Bundled JSON translations are synchronous.
